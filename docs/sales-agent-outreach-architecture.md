# Business Partner — وكيل المبيعات وتوليد العملاء (Sales & Outreach Agent)

التاريخ: 2026-07-22
النطاق: مستند خطة وتصميم معماري فقط (المرحلة صفر). لا يوجد كود في هذه المرحلة —
هذا المستند يُراجَع ويُعتمد قبل بناء أي workflow في n8n.

> **الفكرة باختصار:** وكيل ذكي يجمع داتا الشركات المستهدفة في السعودية من عدة
> مصادر، ينظّفها ويقيّمها، يخزّنها منظّمة في Notion CRM، ثم يشغّل حملات تواصل
> متعددة القنوات (لنكدإن + إيميل تسويقي + واتساب) بتسلسل ذكي — كل ذلك مبني على
> منصة **n8n** المربوطة أصلاً عندنا، ومع احترام صارم للشروط القانونية لكل قناة.

---

## 1. الهدف والنتيجة المطلوبة

بناء "خط إنتاج" (pipeline) للتطوير التجاري يحوّل السوق المفتوح إلى محادثات مبيعات:

```
مصادر الداتا  →  تنظيف وإثراء ودمج  →  Notion CRM (تقييم + تصنيف)  →  حملات متعددة القنوات  →  ردود/اجتماعات
```

**النتيجة النهائية:** قاعدة شركات نظيفة وقابلة للتقسيم (segmentation)، وتسلسل
تواصل تلقائي يوصل الرسالة الصحيحة للشخص الصحيح في القناة الصحيحة، مع تتبّع كامل
للتفاعل داخل CRM واحد.

## 2. الحدود القانونية والامتثال (تُقرأ أولاً — تحكم كل التصميم)

هذي ليست تفاصيل ثانوية؛ هي التي تحدد ما نبنيه وما لا نبنيه.

| القناة/المصدر | القاعدة | ماذا نفعل |
| --- | --- | --- |
| **لنكدإن (scraping مباشر)** | يخالف User Agreement، ويعرّض الحساب للإيقاف ومخاطرة قانونية. | لا نبني سحباً مباشراً بحجم ضخم من حساب شخصي. نستخدم طبقة مصادر مرنة (انظر §4) ونعزل مخاطر لنكدإن في مزوّد مرخّص أو أداة أتمتة رسمية بحدود آمنة. |
| **بيانات شخصية (PDPL السعودي)** | نظام حماية البيانات الشخصية يفرض أساساً نظامياً لمعالجة البيانات، وحق الاعتراض/الحذف. | نجمع بيانات **العمل** (شركة، منصب، إيميل عمل) للأغراض التسويقية المشروعة، نوثّق أساس المعالجة (المصلحة المشروعة / B2B)، ونوفّر رابط إلغاء اشتراك وحذف في كل رسالة. |
| **واتساب (WhatsApp Business API)** | الرسائل التسويقية الباردة بدون موافقة (opt-in) **ممنوعة** وتوقف الرقم. القوالب (templates) لازم تُعتمد مسبقاً. | واتساب يُستخدم فقط بعد opt-in صريح (عميل عبّأ نموذج / تفاعل معنا / وافق). لا نرسل رسائل واتساب باردة لقوائم مسحوبة. |
| **الإيميل التسويقي (CAN-SPAM / أنظمة مكافحة الإزعاج)** | يجب: عنوان مرسِل حقيقي، عنوان فعلي، ورابط إلغاء اشتراك يعمل، وعدم التضليل. | كل حملة تخرج عبر Resend مع footer فيه اسم المنشأة (BP)، السجل التجاري، والعنوان، ورابط unsubscribe فعّال يُحدّث حالة جهة الاتصال في CRM. |
| **حدود المعدّل (rate limits)** | الإرسال الجماعي المفاجئ يحرق السمعة (domain reputation) ويضرب في السبام. | warm-up تدريجي للنطاق، حدود يومية، تقسيم الدُفعات، مراقبة bounce/spam rate. |

**قاعدة ذهبية:** البريد واللنكدإن يمكن أن يبدأا الحوار (outbound بارد ضمن الحدود).
الواتساب قناة **متابعة بعد الموافقة فقط**، ليس قناة بدء بارد.

## 3. المخزون الحالي — نعيد استخدامه ولا نكرّره

ما هو موجود فعلاً في المستودع/المنصة (من قراءة الكود و`api/`):

| الأصل | الموقع | الحالة | دوره في هذا المشروع |
| --- | --- | --- | --- |
| Notion كـ CRM | `NOTION_CRM_DB` (env) + `api/requests.js` | ✅ موجود | قاعدة البيانات المركزية — نوسّع مخططها (§5) |
| إرسال إيميل | Resend عبر `RESEND_API_KEY` + `api/newsletter.js` | ✅ موجود | قناة الإيميل التسويقي |
| جمهور نشرة | `RESEND_AUDIENCE_ID` | ✅ موجود | نموذج مرجعي لإدارة القوائم/الإلغاء |
| n8n | `businesspartnerai.app.n8n.cloud` | ✅ مربوط | محرّك الأتمتة الأساسي لكل الـ pipeline |
| Make.com | مربوط | ✅ | احتياطي/بديل لبعض التكاملات |
| واتساب | `WHATSAPP_URL`, `WHATSAPP_CHANNEL_URL` | ✅ روابط | قناة متابعة بعد opt-in |
| المستشار (شات) | `api/chat.js` + `knowledge.json` | ✅ | مصدر معرفة الخدمات + تأهيل الردود |
| نموذج التقاط عملاء | `api/contact.ts`/`LEAD_WEBHOOK_URL` | ✅ | نقطة opt-in الطبيعية (تُغذّي CRM) |

**لا نكرّر:** CRM ثانٍ، مزوّد إيميل ثانٍ، أو منصة أتمتة ثالثة. كل شيء يمرّ عبر
Notion + Resend + n8n.

## 4. طبقة المصادر المرنة (Data Sources Layer)

بما أنك اخترت الأربعة، نبنيها كـ **مُوحِّدات (adapters) قابلة للتركيب** خلف واجهة
واحدة داخل n8n. كل adapter يُخرج نفس الشكل الموحّد (schema §5) بغض النظر عن المصدر.

```
                     ┌─────────────────────────────────────────┐
                     │   n8n: BP — Company Ingest (موحِّد)        │
   ┌──────────┐      │  يستقبل من أي adapter → ينظّف → يوحّد →     │
   │ Adapter A│─────▶│  يزيل التكرار → يقيّم → يكتب في Notion CRM │
   ├──────────┤      └─────────────────────────────────────────┘
   │ Adapter B│─────▶
   ├──────────┤
   │ Adapter C│─────▶
   ├──────────┤
   │ Adapter D│─────▶
   └──────────┘
```

| Adapter | المصدر | الطريقة | المخاطرة | الأولوية |
| --- | --- | --- | --- | --- |
| **A — مزوّد مرخّص** | Apollo.io / Lusha / Cognism API | استدعاء API رسمي بفلاتر (الدولة=SA، القطاع، الحجم) | منخفضة ✅ | **1 (الأساس)** |
| **B — أتمتة لنكدإن** | PhantomBuster / Sales Navigator export | أداة رسمية بحدود يومية آمنة، تُخرج CSV → webhook | متوسطة ⚠️ | 3 |
| **C — مصادر سعودية عامة** | السجل التجاري/منشآت/مواقع الشركات/دلائل الغرف | جلب موجّه + استخراج بالـ AI | منخفضة/قانونية ✅ | 2 |
| **D — سكرابر مخصص** | صفحات عامة محددة عند الحاجة | آخر خيار، بحدود ضيقة جداً | عالية ⚠️⚠️ | 4 (عند الضرورة فقط) |

**قرار التصميم:** نبدأ بـ A و C (آمنة قانونياً وتغطي أغلب السوق السعودي)،
ونضيف B لاحقاً بضوابط، ونحتفظ بـ D كخيار أخير ومحدود. الواجهة الموحّدة تعني أن
إضافة/إزالة مصدر لا تكسر باقي النظام.

**الإثراء (Enrichment):** بعد الجمع، خطوة تكمّل الناقص (إيميل العمل، حجم الشركة،
الموقع، حساب لنكدإن للشركة) عبر مزوّد الإثراء، وتتحقق من صحة الإيميل
(verification) قبل أي إرسال لتقليل الـ bounce.

## 5. مخطط البيانات في Notion CRM

أربع قواعد مترابطة (relations) داخل Notion:

### 5.1 `Companies` (الشركات)
| الحقل | النوع | ملاحظة |
| --- | --- | --- |
| Name | Title | اسم الشركة |
| Domain | URL | المعرّف الأساسي لإزالة التكرار |
| Sector | Select | القطاع (مقاولات، تجزئة، صناعة...) |
| Size | Select | عدد الموظفين (1-10, 11-50, 51-200, 200+) |
| City / Region | Select | الرياض، جدة، الدمام... |
| LinkedIn URL | URL | صفحة الشركة |
| Source | Multi-select | المصدر (Apollo, Public-SA, Phantom, Manual) |
| ICP Score | Number | تقييم مطابقة العميل المثالي (§6) |
| Stage | Status | New / Enriched / Targeted / Engaged / Won / Lost |
| Contacts | Relation → Contacts | |

### 5.2 `Contacts` (جهات الاتصال)
| الحقل | النوع | ملاحظة |
| --- | --- | --- |
| Full Name | Title | |
| Company | Relation → Companies | |
| Title/Role | Text | المسمّى الوظيفي |
| Work Email | Email | متحقق منه قبل الإرسال |
| Email Status | Select | Valid / Risky / Invalid / Unknown |
| LinkedIn URL | URL | |
| Phone/WhatsApp | Phone | يُستخدم فقط بعد opt-in |
| Consent (WA/Email) | Checkbox | أساس opt-in لكل قناة |
| Unsubscribed | Checkbox | يوقف كل التواصل فوراً |
| Owner | Person | مسؤول المبيعات |

### 5.3 `Campaigns` (الحملات)
اسم، قناة (LinkedIn/Email/WhatsApp)، الجمهور (segment/filter)، القالب، التسلسل
(sequence steps)، الحالة، KPIs.

### 5.4 `Activities` (التفاعلات)
سجل لكل حدث: sent / opened / clicked / replied / bounced / meeting-booked —
مرتبط بجهة الاتصال والحملة. هذا مصدر الحقيقة للتقارير.

> ملاحظة: نعيد استخدام `NOTION_CRM_DB` الحالي كنقطة انطلاق، ونضيف القواعد الناقصة
> ونربطها. لا ننشئ Notion workspace جديد.

## 6. تقييم العميل المثالي (ICP Scoring)

قبل أي حملة، كل شركة تُقيّم آلياً (0–100) لترتيب الأولوية وتجنّب حرق الرسائل:

| المعيار | الوزن | مثال مطابقة عالية |
| --- | --- | --- |
| القطاع | 25 | قطاعات تحتاج إدارة قوى/تأمينات/امتثال |
| الحجم | 25 | 11–200 موظف (يحتاجون إدارة منصات حكومية) |
| الموقع | 15 | مدن التركيز |
| وجود إشارة حاجة | 20 | توظيف نشط، منشأة جديدة، تغيّر مسؤول HR |
| اكتمال الداتا | 15 | إيميل متحقق + جهة قرار |

الشركات بأعلى ICP تدخل التسلسل أولاً. الأقل من عتبة معيّنة تبقى في القائمة
للمراجعة اليدوية أو حملات nurture خفيفة.

## 7. محرّك الحملات متعدد القنوات (Sequencing)

تسلسل موحّد يُدار في n8n، مع قواعد مختلفة لكل قناة حسب §2:

```
اليوم 0  │ LinkedIn: طلب اتصال + رسالة قصيرة مخصّصة (عبر أداة رسمية/يدوي مُعان)
اليوم 2  │ Email #1: تعريف بالقيمة (إدارة منصاتك الحكومية 24/7) + CTA حجز مكالمة
اليوم 5  │ Email #2 (لمن لم يفتح/يرد): زاوية مختلفة + case study مختصرة
اليوم 8  │ LinkedIn: متابعة لمن قَبِل الاتصال
اليوم 12 │ Email #3: عرض/رابط باقات + إلغاء اشتراك واضح
   ↓ عند أي رد إيجابي / حجز / opt-in
WhatsApp │ يُفعَّل فقط الآن (بعد الموافقة) للمتابعة السريعة عبر قالب معتمد
```

**قواعد الإيقاف التلقائي:** أي رد بشري، أو نقرة "غير مهتم"، أو unsubscribe →
يوقف التسلسل فوراً ويحدّث `Stage`/`Unsubscribed` في CRM.

**التخصيص بالـ AI:** خطوة في n8n تستدعي Claude (نفس مفتاح `ANTHROPIC_API_KEY`
المستخدم في `api/chat.js`) لتوليد سطر افتتاحي مخصّص لكل شركة من داتا CRM، مع
حراسة صارمة ضد الادعاءات المضلّلة.

## 8. workflows الـ n8n المطلوب بناؤها

| # | Workflow | الدور | مشغّل (trigger) |
| --- | --- | --- | --- |
| W1 | `BP — Company Ingest` | يستقبل من كل adapter، ينظّف/يوحّد/يزيل التكرار، يكتب Companies+Contacts | Webhook + Schedule |
| W2 | `BP — Enrich & Verify` | يكمّل الناقص + يتحقق من الإيميل + يحدّث Email Status | يعمل بعد W1 |
| W3 | `BP — ICP Scoring` | يحسب ICP Score ويحدّث Stage=Targeted للمؤهلين | Schedule يومي |
| W4 | `BP — Email Sequencer` | يرسل تسلسل الإيميل عبر Resend، يسجّل Activities، يحترم الحدود والإلغاء | Schedule + حالة |
| W5 | `BP — LinkedIn Outreach` | يجهّز مهام لنكدإن (عبر أداة رسمية/قائمة يدوية مُعانة) ويتتبّع القبول/الرد | Schedule |
| W6 | `BP — WhatsApp Follow-up` | يرسل قوالب معتمدة **فقط** لمن عنده Consent=true بعد رد إيجابي | حدث opt-in |
| W7 | `BP — Reply & Suppression` | يلتقط الردود/الإلغاء/الـ bounce ويوقف التسلسل ويحدّث CRM | Webhook/IMAP |
| W8 | `BP — Sales Dashboard` | يجمّع KPIs أسبوعياً ويرسل تقرير (يعيد استخدام نمط `api/newsletter.js`) | Cron أسبوعي |

هذي أسماء مقترحة تتبع اصطلاح `BP — …` الموجود في n8n (مثل `BP — Website ATS Intake`).

## 10. مؤشرات الأداء (KPIs)

قمع من أعلى لأسفل، كلها تُحسب من قاعدة `Activities`:

- عدد الشركات المجمّعة / المؤهّلة (ICP فوق العتبة)
- معدّل صحة الإيميل (verified %) و bounce rate (هدف < 3%)
- Email: open rate, reply rate, meeting-booked rate
- LinkedIn: acceptance rate, reply rate
- WhatsApp: تفاعل ما بعد opt-in
- التحويل النهائي: عملاء (Stage=Won) وقيمتهم

## 11. المخاطر والتخفيف

| الخطر | التخفيف |
| --- | --- |
| إيقاف حساب لنكدإن | لا سحب جماعي؛ أداة رسمية بحدود؛ الاعتماد على مزوّد مرخّص كأساس |
| حرق سمعة نطاق الإيميل | warm-up تدريجي، حدود يومية، تحقق مسبق، فصل نطاق فرعي للإرسال (مثلاً `mail.` أو نطاق منفصل) |
| إيقاف رقم واتساب | opt-in صارم، قوالب معتمدة فقط، لا رسائل باردة |
| مخالفة PDPL | أساس معالجة موثّق، بيانات عمل فقط، إلغاء/حذف فوري عند الطلب |
| داتا مكررة/متسخة | إزالة تكرار بالـ Domain، تطبيع، تحقق قبل الإرسال |

## 13. الخطوة التالية (تحتاج قرارك)

عند اعتماد هذا المستند، أبدأ بـ **P1**:
1. أبني مخطط الـ 4 قواعد في Notion وأربطها.
2. أبني workflow `BP — Company Ingest` في n8n بالواجهة الموحّدة.
3. أوصّل أول adapter (المزوّد المرخّص) — يحتاج منك اسم المزوّد ومفتاح API.

**أسئلة تبقى مفتوحة قبل P1:**
- أي مزوّد داتا مرخّص نستخدم؟ (Apollo أرخص وأشمل للبداية / Lusha / Cognism)
- ما القطاعات ومدن التركيز والحجم المستهدف بالضبط (لضبط ICP)؟
- هل نستخدم نطاق الإرسال الحالي `businesspartner.sa` أم نطاق فرعي منفصل لحماية السمعة؟
